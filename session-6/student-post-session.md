## What We Built

Session 6 introduced **admin-only operations** into the CRM. Until now, user accounts were created by a seed script and leads were added one at a time through a dialog. In this session, we built the infrastructure that a real team needs on day one: invite-based user onboarding, role management, soft deletion, and bulk lead import. Every route we shipped is protected by `authenticateUser([Role.ADMIN])` — admin operations get their own security boundary.

By the end of Session 6, the `whispyr-academy-crm` repo had:

- **Resend email client** — a shared `src/lib/resend.ts` module wrapping the Resend SDK for transactional email delivery (invites today, more templates later)
- **Admin service module** — the sixth service to follow the 5-file pattern (`schema`, `db`, `helpers`, `service`, `index`) covering user creation, listing, role editing, deactivation, and reactivation
- **`AdminServiceError`** — a new domain error class wired into `handleRouteError` so admin-layer failures surface as proper HTTP status codes
- **Magic link invite flow** — `createUser` orchestrates four systems: check-for-duplicate → create Supabase Auth user → create Prisma `Profile` → send invite email via Resend with a `generateLink({ type: "magiclink" })` URL
- **HTML email template** — an inline-styled, Gmail/Outlook-safe invite email generated by `generateInviteEmailHTML` in `src/services/admin/helpers.ts`
- **Soft deletion** — `deactivateUser` and `reactivateUser` flip the existing `isActive` column on `Profile` instead of deleting rows, preserving every foreign-key relationship and audit trail
- **Admin API routes** — `GET/POST /api/admin/users`, `PUT /api/admin/users/[id]`, `POST /api/admin/users/[id]/deactivate`, and `POST /api/admin/users/[id]/reactivate`, all gated by `authenticateUser([Role.ADMIN])`
- **Self-protection checks** — an admin cannot change their own role or deactivate themselves; the service layer throws 400 before any database write happens
- **TanStack Query hooks for users** — `useUsers`, `useCreateUser`, `useUpdateUser`, `useDeactivateUser`, and `useReactivateUser` in `src/lib/tanstack/useUsers.ts`, all keyed on `["admin", "users"]` and invalidating on success
- **Admin users page** — `/admin/users` server component that performs a server-side admin role check and redirects non-admins, then renders the client-side `UsersPageClient` with `UsersTable`, `CreateUserDialog`, and a per-row `UserActions` dropdown
- **Role and status badges** — color-coded badges for ADMIN/MANAGER/AGENT role and Active/Inactive status on every user row
- **Import/Export service module** — a new `src/services/import-export/` module with `csvLeadRowSchema`, `parseCSV` (papaparse), `validateRows` (per-row Zod with error collection), `buildCSVString` (RFC 4180 escaping), `generateTemplateCSV`, and `processImport`
- **CSV import API route** — `POST /api/admin/import` that re-validates every row on the server with the SAME Zod schema the client used, then calls `processImport` to create leads
- **CSV importer UI** — `src/components/admin/CsvImporter.tsx` with a four-state machine (idle → parsing → validated → importing), drag-and-click upload, invalid-row table, count badges, and a one-click template download
- **Admin import page** — `/admin/import` server component with the same admin auth check as the users page
- **`useImport` mutation hook** — fires the import and invalidates `["leads"]` on success so the main leads table refetches automatically
- **Sidebar nav + proxy matcher updates** — `/admin/users` and `/admin/import` added to the sidebar (admin-only), and the middleware matcher extended to cover `/admin/*` routes
- **CSV template files** — `leads-import-template.csv` (four columns, two example rows) and `sample_contacts.csv` (five example contacts in Egyptian E.164 format) checked into the repo for students to test against

> **Note:** We built user management end-to-end (create + edit + deactivate + reactivate with magic link emails) and the full CSV import pipeline. We did **NOT** build CSV export, admin layout tabs, pagination for the users table, the valid-row preview table, the re-send-invite action, the middleware `isActive` check extension, or bulk operations. ### Phase 1: Security Mental Model + Resend Client Setup

**Concepts covered:** Service Role Key vs. anon key, server-only modules, transactional email providers, JIT package installation.

**What we did:**

Set up the two external services Session 6 depends on: the Supabase Admin client (already created during initial setup, used for the first time today) and the Resend email client.

**Reference file:** `src/lib/supabase/admin.ts` (already existed)

This file was scaffolded during initial setup but never imported until now:

```typescript
import { createClient } from "@supabase/supabase-js";

const admin = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!, // ← NOT the anon key
  {
    auth: { autoRefreshToken: false, persistSession: false },
  },
);

export default admin;
```

**Key concept — Service Role Key bypasses Row Level Security:**

The Service Role Key is the root key for your Supabase project. It ignores Row Level Security policies, bypasses rate limits, and can create users, delete users, set email confirmation, generate magic links — anything. If this key leaks to the browser, an attacker can do literally anything to your database.

Two rules:

1. **Server-only.** Never import `src/lib/supabase/admin.ts` from a client component. Next.js will happily bundle it into the browser if you do, and the key will ship in your JavaScript.
2. **Never `NEXT_PUBLIC_`.** The env var is `SUPABASE_SERVICE_ROLE_KEY` — no public prefix. The `NEXT_PUBLIC_SUPABASE_URL` is fine because the URL is discoverable anyway.

**Why `autoRefreshToken: false, persistSession: false`?**

The admin client is used to perform privileged operations — not to log in as a user. It has no session to refresh and nothing to persist. Disabling both prevents the client from trying to store tokens in a non-existent browser storage.

**Reference file:** `src/lib/resend.ts`

JIT-installed during the session:

```bash
npm install resend
```

```typescript
import { Resend } from "resend";

export const resend = new Resend(process.env.RESEND_API_KEY);
```

**Key concept — Transactional email is a separate system:**

Sending email from your app server is a bad idea — you hit IP reputation issues, deliverability problems, and spam filters within hours. Transactional email providers (Resend, Postmark, SendGrid) maintain warm IP pools, handle DKIM/SPF signing, and give you delivery analytics. We use Resend because the SDK is small, the free tier is generous, and the API surface is minimal: `resend.emails.send({ from, to, subject, html })`.

### Phase 2: Admin Service Module — Schema + DB

**Concepts covered:** The 5-file service pattern (sixth time), Zod schemas for admin operations, select clauses in DB queries, soft deletion queries.

**What we did:**

Built the schema and DB layers for the admin service following the same structure from the lead, activity, reminder, notification, and AI services.

**Service folder structure:**

```text
src/services/admin/
  schema.ts      ← Zod schemas for user create + update
  db.ts          ← Prisma queries: find, list, create, update, deactivate, reactivate
  helpers.ts     ← HTML email template generation
  service.ts     ← createUser, listUsers, getUserById, updateUserById, deactivateUser, reactivateUser
  index.ts       ← Barrel exports
```

**Reference file:** `src/services/admin/schema.ts`

```typescript
import { Role } from "@/generated/prisma/enums";
import { z } from "zod";

export const createUserSchema = z.object({
  email: z.email().transform((email) => email.toLowerCase().trim()),
  name: z.string().min(1).max(100),
  role: z.enum(Role),
});

export type CreateUserSchema = z.infer<typeof createUserSchema>;

export const updateUserSchema = z.object({
  name: z.string().min(1).max(100).optional(),
  role: z.enum(Role).optional(),
});

export type UpdateUserSchema = z.infer<typeof updateUserSchema>;
```

**Key design — `.transform()` on email:**

Emails are case-insensitive by RFC 5321. Storing them lowercased at validation time means the DB layer never has to guess, `findUnique({ where: { email } })` works predictably, and duplicate detection stays consistent. The `.trim()` catches the trailing whitespace users paste in from spreadsheets.

**Key design — Two schemas, not four:**

We collapsed "update role" and "update name" into a single `updateUserSchema` with both fields optional. The route calls it with whichever fields are present. Deactivate and reactivate don't need a schema at all — they're ID-only operations exposed as dedicated sub-routes (see Phase 3).

**Reference file:** `src/services/admin/db.ts`

```typescript
export async function dbFindUserByEmail(email: string) {
  return await prisma.profile.findUnique({
    where: { email },
    select: {
      id: true,
      email: true,
      name: true,
      role: true,
      isActive: true,
    },
  });
}

export async function dbCreateProfile(data: {
  id: string;
  email: string;
  name: string;
  role: Role;
}) {
  return prisma.profile.create({
    data: {
      id: data.id, // Must match Supabase Auth user ID
      email: data.email,
      name: data.name,
      role: data.role,
      isActive: true,
    },
    select: {
      id: true,
      email: true,
      name: true,
      role: true,
      isActive: true,
      createdAt: true,
    },
  });
}

export async function dbDeactivateUser(userId: string) {
  return prisma.profile.update({
    where: { id: userId },
    data: { isActive: false },
    select: {
      id: true, email: true, name: true, role: true, isActive: true,
    },
  });
}
```

**Key design — Every query has a `select` clause:**

We follow the Whispyr code-quality rule: every Prisma query explicitly selects the fields it returns. This means the return type is always known, you never leak sensitive columns by accident, and the SQL the database actually runs is small.

**Key design — `dbCreateProfile` takes `id` as input:**

The `Profile` model's `id` is NOT auto-generated — it must match the Supabase Auth `user.id`. This is the **bridging pattern**: two systems, one identity. Supabase Auth owns authentication (password hashes, sessions, email verification); our Prisma `Profile` owns business data (role, name, isActive). They're joined by sharing a UUID. If you let Prisma generate its own UUID, the two systems drift and authentication breaks.

**Key design — Soft deletion via `isActive`:**

```typescript
data: { isActive: false }
```

We never DELETE rows. If an agent leaves the company, their `Profile` stays — every lead they touched, every activity they logged, every reminder they created still points to them. Flipping `isActive` to `false` achieves the business outcome (they can't log in) without breaking referential integrity. The `authenticateUser` utility already rejects users with `isActive: false`, so deactivation takes effect on their very next request.

### Phase 3: Admin Service Module — Helpers + Service + Index

**Concepts covered:** HTML email template engineering, multi-system orchestration, custom error classes, self-protection rules, graceful degradation.

**What we did:**

Built the business logic layer: the `createUser` orchestrator (the hardest function in the session), the role/deactivation operations with self-protection checks, and the invite email generator.

**Reference file:** `src/services/admin/helpers.ts`

```typescript
export function generateInviteEmailHTML(
  name: string,
  magicLink: string,
): string {
  return `
<!DOCTYPE html>
<html>
  <body>
    <div class="container" style="max-width: 560px; ...">
      <h1>Welcome to CRM Pro</h1>
      <p>Hi ${name},</p>
      <p>An administrator has created an account for you. Click the button below to sign in:</p>
      <a href="${magicLink}" style="background-color: #18181b; color: #ffffff; ...">
        Sign In to CRM Pro
      </a>
      <p>Or copy and paste this link into your browser:</p>
      <div style="word-break: break-all; ...">${magicLink}</div>
      <p><strong>This link expires in 24 hours</strong> and can only be used once.</p>
    </div>
  </body>
</html>`.trim();
}
```

**Key concept — Email HTML is not web HTML:**

Gmail, Outlook, and Apple Mail each strip or rewrite parts of your HTML. The `<style>` block may be removed entirely. JavaScript is always stripped. External CSS is ignored. You must:

1. **Inline every style** directly on the element. We duplicate styles in both `<style>` and `style=""` attributes — the `<style>` block covers modern clients, the inline styles are the fallback.
2. **Provide a fallback plaintext link.** Outlook sometimes renders buttons as raw HTML or strips them entirely. The plaintext link below the button means the email always works.
3. **Keep it narrow (max ~560px).** Email clients have no concept of responsive design — they just show what you give them.

For a small app with 1–2 templates, inline HTML in a helper is fine. If we reach 10+ templates, we'll switch to React Email or MJML.

**Reference file:** `src/services/admin/service.ts` — `createUser`

```typescript
export async function createUser(data: CreateUserSchema) {
  // Step 1: Check for duplicate email
  const existingUser = await dbFindUserByEmail(data.email);
  if (existingUser) throw new AdminServiceError("User already exists", 409);

  // Step 2: Create Supabase Auth user with a temp password
  const tempPassword = uuidv4().toString();
  const { data: authData, error: authError } =
    await supabaseAdmin.auth.admin.createUser({
      email: data.email,
      password: tempPassword,
      email_confirm: true,
    });

  if (authError || !authData.user) {
    throw new AdminServiceError("Failed to create Supabase Auth user", 500);
  }

  // Step 3: Create Profile in Prisma (shares the Auth user ID)
  const profile = await dbCreateProfile({
    id: authData.user.id,
    email: data.email,
    name: data.name,
    role: data.role,
  });

  // Step 4: Generate a magic link and send the invite email
  const { data: magicLinkData, error: magicLinkError } =
    await supabaseAdmin.auth.admin.generateLink({
      type: "magiclink",
      email: data.email,
    });

  if (magicLinkError || !magicLinkData?.properties?.action_link) {
    console.error("Error generating magic link:", magicLinkError);
    return profile; // ← Graceful degradation: user is still created
  }

  const magicLink = magicLinkData.properties.action_link;

  await resend.emails.send({
    from: "onboarding@resend.dev",
    to: data.email,
    subject: "You're invited to CRM Pro",
    html: generateInviteEmailHTML(data.name, magicLink),
  });

  return profile;
}
```

**Key concept — The four-step orchestration:**

`createUser` coordinates three independent systems in a specific order:

```
┌─────────────────────────────────────────────────────────────────┐
│  1. dbFindUserByEmail  →  Prisma (check duplicates)             │
│  2. supabaseAdmin.auth.admin.createUser  →  Supabase Auth       │
│  3. dbCreateProfile(id: authData.user.id)  →  Prisma (bridge)   │
│  4. generateLink + resend.emails.send  →  Supabase + Resend     │
└─────────────────────────────────────────────────────────────────┘
```

Step 2 MUST come before step 3 because step 3 needs the UUID that Supabase generates. The order is not a style choice — it's a data dependency.

**Key concept — Graceful degradation on email failure:**

If the magic-link generation or email send fails, we log the error and return the profile anyway. We do NOT throw:

```typescript
if (magicLinkError || !magicLinkData?.properties?.action_link) {
  console.error("Error generating magic link:", magicLinkError);
  return profile; // user is created, email just didn't go out
}
```

The reasoning: the user account is valid. The database is consistent. An admin can re-trigger the invite email later (the "Resend invite" homework task). Failing hard here would force the admin to delete the user and start over, which is worse UX.

**Key concept — `email_confirm: true` skips the verification step:**

By default Supabase sends its own "confirm your email" message when you create a user. We set `email_confirm: true` to mark the email as already verified — because the admin is vouching for the user. This also means we control the email content (ours, not Supabase's default).

**Key concept — `tempPassword` is never shown to the user:**

Supabase Auth requires a password to create a user. We pass a random UUID as a throwaway password. The user never sees it — they log in via the magic link, then can set a real password later in their profile settings.

**Reference file:** `src/services/admin/service.ts` — self-protection checks

```typescript
export async function updateUserById(
  adminId: string,
  targetUserId: string,
  data: UpdateUserSchema,
) {
  if (adminId === targetUserId) {
    throw new AdminServiceError("You cannot change your own role", 400);
  }
  const target = await dbFindUserById(targetUserId);
  if (!target) throw new AdminServiceError("User not found", 404);
  return dbUpdateUser(targetUserId, data);
}

export async function deactivateUser(adminId: string, targetUserId: string) {
  if (adminId === targetUserId) {
    throw new AdminServiceError("You cannot deactivate yourself", 400);
  }
  const target = await dbFindUserById(targetUserId);
  if (!target) throw new AdminServiceError("User not found", 404);
  if (!target.isActive) {
    throw new AdminServiceError("User is already deactivated", 400);
  }
  return dbDeactivateUser(targetUserId);
}
```

**Software engineering principle — Self-protection checks:**

An admin accidentally demoting themselves from ADMIN to AGENT locks the entire team out of admin features until someone hits the database directly. An admin deactivating themselves has the same effect — plus they lose their session on the next request. These are recoverable but expensive mistakes. Refusing the operation with a 400 is cheaper than writing a recovery runbook.

**Reference file:** `src/services/admin/service.ts` — `AdminServiceError`

```typescript
export class AdminServiceError extends Error {
  constructor(
    message: string,
    public statusCode: number,
  ) {
    super(message);
    this.name = "AdminServiceError";
  }
}
```

Same shape as `LeadServiceError` and `NotificationServiceError`. Register it in `handleRouteError.ts`:

```typescript
if (
  error instanceof AuthenticationError ||
  error instanceof LeadServiceError ||
  error instanceof NotificationServiceError ||
  error instanceof AdminServiceError  // ← new
) {
  return NextResponse.json(
    { error: error.message },
    { status: error.statusCode },
  );
}
```

### Phase 4: Admin API Routes — Protecting the Boundary

**Concepts covered:** Role-based route authentication, dedicated action routes vs. PATCH with body, thin route handlers.

**What we did:**

Built four route files covering list, create, update, deactivate, and reactivate.

**Reference file:** `src/app/api/admin/users/route.ts`

```typescript
// GET /api/admin/users — list all users
export async function GET() {
  try {
    await authenticateUser([Role.ADMIN]);
    const users = await AdminService.user.list();
    return NextResponse.json({ success: true, data: users });
  } catch (error) {
    return handleRouteError(error);
  }
}

// POST /api/admin/users — create a user with invite
export async function POST(request: NextRequest) {
  try {
    await authenticateUser([Role.ADMIN]);
    const body = await request.json();
    const data = AdminSchema.user.create.parse(body);
    const user = await AdminService.user.create(data);
    return NextResponse.json({ success: true, data: user }, { status: 201 });
  } catch (error) {
    return handleRouteError(error);
  }
}
```

**Key concept — `authenticateUser([Role.ADMIN])` as a non-negotiable boundary:**

The `authenticateUser` utility (from Session 5's middleware work) now takes an array of allowed roles. If the caller isn't authenticated it throws 401; if their role isn't in the allowed list it throws 403. A single line guarantees that only admins can reach the business logic below.

This runs on the server, not the middleware. The middleware does a coarse session check (is the cookie valid?); route-level `authenticateUser` does the fine-grained role check (is this user an admin?). Middleware is about page access; route authentication is about API access.

**Reference file:** `src/app/api/admin/users/[id]/route.ts`

```typescript
export async function PUT(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  try {
    const profile = await authenticateUser([Role.ADMIN]);
    const { id } = await params;
    const body = await request.json();
    const data = AdminSchema.user.update.parse(body);
    const user = await AdminService.user.update(profile.id, id, data);
    return NextResponse.json({ success: true, data: user });
  } catch (error) {
    return handleRouteError(error);
  }
}
```

**Key concept — Passing `profile.id` for self-protection:**

The service's `updateUserById` needs to know **who is calling** to enforce the "cannot edit yourself" rule. The route grabs the profile from `authenticateUser` and passes its `id` through. The service does the comparison.

**Reference files:** `src/app/api/admin/users/[id]/deactivate/route.ts` and `.../reactivate/route.ts`

```typescript
// POST /api/admin/users/[id]/deactivate
export async function POST(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  try {
    const profile = await authenticateUser([Role.ADMIN]);
    const { id } = await params;
    const user = await AdminService.user.deactivate(profile.id, id);
    return NextResponse.json({ success: true, data: user });
  } catch (error) {
    return handleRouteError(error);
  }
}
```

**Key design — Dedicated action routes over PATCH with body:**

We put deactivate and reactivate on their own sub-routes (`.../[id]/deactivate` and `.../[id]/reactivate`) rather than stuffing them into a PATCH with `{ action: "deactivate" }` in the body. Same reason as the reminder mark-read route in Session 4:

- The URL is self-documenting. `POST /api/admin/users/abc/deactivate` explains itself.
- Each route has one action, one permission check, one response shape.
- If we add snooze, lock, or invite-resend later, each gets its own path.
- PATCH-with-action turns into a giant switch statement with scattered authorization logic.

The trade-off: more files. In a CRM with five admin actions, that's fine. At 50 actions we'd revisit.

### Phase 5: TanStack Hooks for User Management

**Concepts covered:** Query keys for admin resources, mutation invalidation patterns, standardized `User` type.

**What we did:**

Built five TanStack hooks — one query, four mutations — in `src/lib/tanstack/useUsers.ts`. Every mutation invalidates the same `["admin", "users"]` key so the users table updates the moment any action completes.

**Reference file:** `src/lib/tanstack/useUsers.ts`

```typescript
export type User = {
  id: string;
  email: string;
  name: string;
  role: Role;
  isActive: boolean;
  createdAt: Date;
};

export function useUsers() {
  return useQuery<User[]>({
    queryKey: ["admin", "users"],
    queryFn: async () => {
      const { data } = await api.get("/admin/users");
      return data.data;
    },
  });
}

export function useCreateUser() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: async (user: CreateUserSchema): Promise<User> => {
      const { data } = await api.post("/admin/users", user);
      return data.data;
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["admin", "users"] });
    },
  });
}

export function useUpdateUser(id: string) {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: async (user: UpdateUserSchema): Promise<User> => {
      const { data } = await api.put(`/admin/users/${id}`, user);
      return data.data;
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["admin", "users"] });
    },
  });
}

export function useDeactivateUser(id: string) {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: async (): Promise<User> => {
      const { data } = await api.post(`/admin/users/${id}/deactivate`);
      return data.data;
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["admin", "users"] });
    },
  });
}

export function useReactivateUser(id: string) { /* mirror of deactivate */ }
```

**Key design — Nested query key `["admin", "users"]`:**

The first segment scopes the key to admin resources. Later, `["admin", "audit-log"]` or `["admin", "billing"]` will join the namespace. Flat keys (`["users"]`) clash with the CRM's existing user references from the assignment dropdown; nesting prevents collisions.

**Key design — No `id` parameter needed on `useCreateUser`:**

`useCreateUser` doesn't need to be keyed by an ID because it always posts to the same URL. `useUpdateUser`, `useDeactivateUser`, and `useReactivateUser` all take `id` as a hook argument because they operate on a specific user resource. The ID becomes part of the mutation closure.

**Reference file:** `src/app/(protected)/admin/users/page.tsx`

```typescript
export default async function AdminUsersPage() {
  const supabase = await createSupabaseServerClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect("/login");

  const profile = await prisma.profile.findUnique({ where: { id: user.id } });

  if (!profile || !profile.isActive) {
```

**Key concept — Server Component as the admin gate:**

Compare to a client-only check where the page ships first, then the browser runs JavaScript and redirects: that's slower, leaks the page structure, and is easy to bypass by disabling JS.

**Key concept — Three layers of defense:**

1. **Middleware** (Session 5) — checks that a session exists, blocks unauthenticated traffic at the edge.
2. **Page-level auth check** (this Server Component) — confirms role and active status before rendering.
3. **API-level `authenticateUser([Role.ADMIN])`** — blocks non-admins from calling the underlying routes.

If layer 1 breaks, layer 2 catches. If layer 2 is bypassed (direct API call), layer 3 catches. **Defense in depth** — the same principle from Sessions 2–5, applied at every boundary.

```typescript
export const csvLeadRowSchema = z.object({
  phone: z
    .string()
    .min(1, "Phone is required")
    .regex(
      /^\+?[1-9]\d{1,14}$/,
      "Phone must be a valid E.164 number (e.g., +15551234567)",

  rows: z.array(csvLeadRowSchema),

export interface RowValidationResult {

  valid: boolean;
  data: CSVLeadRow | null;
  errors: Record<string, string[]> | null;
}

  importedCount: number;
  totalProcessed: number;
  errors: string[];

```

**Key design — `.optional().or(z.literal("")).transform()`:**

CSV cells come through as strings. An optional field in a CSV is usually an **empty string**, not `undefined`. Three stacked Zod methods handle this:

1. `.optional()` — allow the field to be missing entirely
2. `.or(z.literal(""))` — also accept the literal empty string
3. `.transform((val) => (val === "" ? undefined : val))` — normalize `""` to `undefined` so the downstream code doesn't have to check both

This is the canonical pattern for optional CSV fields.

**Key design — E.164 phone regex:**

```
/^\+?[1-9]\d{1,14}$/
```

E.164 is the international phone number standard: optional `+`, 1–15 digits, first digit must be 1–9 (no leading zero). This catches `+20123456789`, rejects `+0123`, rejects `abc`, and accepts US, Egyptian, and international formats uniformly. Real-world phone parsing is harder (libphonenumber exists for a reason), but E.164 is 95% of what a CRM needs.

**Key design — `RowValidationResult` carries the row number:**

```typescript
rowNumber: index + 2  // +1 for header, +1 for 0-indexing
```

When we show validation errors to the user, they need to know **which row** broke. We preserve the CSV row number (1-indexed, header is row 1, data starts at row 2) so the admin can open the file in a spreadsheet and fix the exact row.

**Reference file:** `src/services/import-export/helpers.ts` — `parseCSV`

```typescript
export function parseCSV(file: File) {
  return new Promise<Record<string, string>[]>((resolve, reject) => {
    Papa.parse(file, {
      header: true,
      skipEmptyLines: true,
      dynamicTyping: false,
      complete: (results) => {
        if (results.errors.length > 0) {
          console.warn("CSV parsing warnings:", results.errors);
        }
        resolve(results.data as Record<string, string>[]);
      },
      error: (error) => reject(new Error("Failed to parse CSV file")),
    });
  });
}
```

**Key concept — Why papaparse:**

You might think "CSV is just commas and newlines" — and you'd be wrong. RFC 4180 says:

- Fields containing commas must be wrapped in double quotes: `"Smith, Jane"`
- Fields containing double quotes escape them by doubling: `"She said ""hi"""`
- Fields containing newlines must be wrapped in double quotes
- Line endings can be `\n`, `\r\n`, or (rarely) `\r`
- Leading/trailing whitespace is ambiguous

Writing a correct CSV parser is a week of work and edge cases. Papaparse is 40KB and handles all of it. Always use a library for CSV.

**Papa.parse options we use:**

- `header: true` — first row is treated as column names, subsequent rows become `{ phone: "...", name: "..." }` objects
- `skipEmptyLines: true` — blank lines between rows don't turn into empty records
- `dynamicTyping: false` — DON'T auto-convert "123" to a number. Phone numbers look numeric but need to stay strings (leading zeros, `+` signs)

**Reference file:** `src/services/import-export/helpers.ts` — `validateRows`

```typescript
export function validateRows(
  rows: Record<string, string>[],
): RowValidationResult[] {
  return rows.map((row, index) => {
    const result = csvLeadRowSchema.safeParse(row);

    if (result.success) {
      return {
        rowNumber: index + 2,
        valid: true,
        data: result.data,
        errors: null,
      };
    }

    return {
      rowNumber: index + 2,
      valid: false,
      data: null,
      errors: result.error.flatten().fieldErrors as Record<string, string[]>,
    };
  });
}
```

**Key concept — `safeParse` instead of `parse`:**

`parse` throws on validation failure. `safeParse` returns a `{ success, data | error }` discriminated union. When validating 500 rows, we do NOT want one bad row to throw and abort the whole batch. `safeParse` lets us collect all errors and show them to the admin at once.

**Key concept — `result.error.flatten().fieldErrors`:**

Zod errors are nested by default. `.flatten()` collapses them into `{ fieldErrors: { phone: ["Phone is required"], email: ["Invalid email"] } }` — perfect for rendering a table of errors by field.

**Reference file:** `src/services/import-export/helpers.ts` — `buildCSVString` and `escapeField`

```typescript
export function buildCSVString(
  data: Record<string, unknown>[],
  headers?: string[],
): string {
  if (data.length === 0) return "";
  const columns = headers ?? Object.keys(data[0]);
  const headerRow = columns.map(escapeField).join(",");
  const dataRows = data.map((row) =>
    columns.map((col) => escapeField(String(row[col] ?? ""))).join(","),
  );
  return [headerRow, ...dataRows].join("\n");
}

function escapeField(value: string): string {
  // RFC 4180: if field contains comma, quote, or newline → wrap in quotes
  //           if field contains a quote → double it ("" inside "")
  if (value.includes(",") || value.includes('"') || value.includes("\n")) {
    return `"${value.replace(/"/g, '""')}"`;
  }
  return value;
}
```

We wrote our own builder (instead of using `Papa.unparse`) to make the escaping rules explicit. In production, Papa.unparse is fine — understand the rules once, then use the library.

### Phase 8: CSV Import Service — Business Logic + Route

**Concepts covered:** Double validation pattern, service-to-service calls, per-row error recovery, reusing existing services.

**What we did:**

Built `processImport` in the service layer, the DB helper for assignee resolution, and the API route that re-validates and delegates.

**Reference file:** `src/services/import-export/service.ts`

```typescript
export async function processImport(
  rows: CSVLeadRow[],
  importerProfile: Profile,
): Promise<ImportSummary> {
  let importedCount = 0;
  const errors: string[] = [];

  for (const row of rows) {
    try {
      // Resolve assignee by email (if provided)
      let assignedToId: string | undefined;
      if (row.assigneeEmail) {
        const assignee = await dbFindProfileByEmail(row.assigneeEmail);

        if (!assignee) {
          errors.push(`Row (${row.phone}): Assignee "${row.assigneeEmail}" not found`);
          continue;
        }
        if (!assignee.isActive) {
          errors.push(`Row (${row.phone}): Assignee "${row.assigneeEmail}" is deactivated`);
          continue;
        }
        assignedToId = assignee.id;
      }

      // Create lead (reuses LeadService — activities get logged automatically)
      const lead = await LeadService.createLead(importerProfile, row);

      // Reassign if assignee was specified
      if (assignedToId) {
        await LeadService.updateLead(importerProfile, lead.id, { assignedToId });
      }

      importedCount++;
    } catch (error) {
      const message = error instanceof Error ? error.message : "Unknown error";
      errors.push(`Row (${row.phone}): ${message}`);
    }
  }

  return { importedCount, totalProcessed: rows.length, errors };
}
```

**Key concept — Service-to-service calls (third time):**

`processImport` calls `LeadService.createLead` and `LeadService.updateLead`. This is the same pattern from Sessions 3 (lead calls activity) and 4 (reminder calls notification). The import service doesn't know how to create a lead — it delegates to the lead service, which knows how to:

- Validate the lead payload against the lead schema
- Write the Lead row
- Log a `LEAD_CREATED` activity on the timeline
- Log an `ASSIGNMENT_CHANGE` activity when `updateLead` is called with a new `assignedToId`

Every imported lead ends up in the database with a full audit trail — because we reuse the existing lead service instead of writing a new "bulk create" path that bypasses the activity logging.

**Key concept — Per-row error recovery:**

The whole loop is wrapped in `try/catch`. If row 347 throws (duplicate phone, missing assignee, etc.), we log the error and continue to row 348. The admin gets a summary: "Imported 498 rows. 2 errors: row 347 duplicate phone, row 412 assignee not found." This is far better UX than "import failed on row 347" and rolling back 346 successful inserts.

**Trade-off:** We don't get all-or-nothing atomicity. For bulk import, partial success with a clear error report is almost always the right call. If a business requirement demands atomicity, wrap the whole loop in `prisma.$transaction` and throw on the first error.

**Key design — Assignee resolution by email, not ID:**

CSV files are edited by humans. Asking a sales manager to look up UUIDs is pointless. We accept `assigneeEmail` in the CSV, look up the profile ourselves, and reject the row (with a clear error) if the email doesn't match an active user.

**Software engineering principle — Double validation:**

**Because the client can lie.** Anyone can open devtools, call `api.post("/admin/import", { rows: [{ ... malicious payload ... }] })`, and skip the UI entirely. The server cannot trust client-side validation for anything. The client schema is for UX (instant feedback); the server schema is the **security boundary**.

### Phase 9: CSV Importer UI — The State Machine

**Concepts covered:** Finite state machines for UI flow, client-side file handling, `URL.createObjectURL` for downloads, disabled-button guards.

**What we did:**

Built `src/components/admin/CsvImporter.tsx` — a client component with four explicit states: `idle`, `parsing`, `validated`, `importing`. Each state determines which UI elements render, which buttons are enabled, and what the user can do next.

**Reference file:** `src/components/admin/CsvImporter.tsx`

```typescript
type ImportState = "idle" | "parsing" | "validated" | "importing";

export function CSVImporter() {
  const [state, setState] = useState<ImportState>("idle");
  const [results, setResults] = useState<RowValidationResult[]>([]);
  const { mutate: importCSV, isPending } = useImport();

  async function handleFileSelect(e: React.ChangeEvent<HTMLInputElement>) {
    const file = e.target.files?.[0];
    if (!file) return;

    setState("parsing");
    try {
      const rows = await parseCSV(file);             // step 1: papaparse
      const validationResults = validateRows(rows);   // step 2: Zod
      setResults(validationResults);
      setState("validated");

      const validCount = validationResults.filter((r) => r.valid).length;
      const invalidCount = validationResults.filter((r) => !r.valid).length;
      toast.info(`Parsed ${validationResults.length} rows: ${validCount} valid, ${invalidCount} invalid`);
    } catch {
      toast.error("Failed to parse CSV file. Check the format and try again.");
      setState("idle");
    }

    e.target.value = ""; // reset so same file can be re-selected
  }

  function handleImport() {
    const validRows = results
      .filter((r) => r.valid && r.data !== null)
      .map((r) => r.data!);

    if (validRows.length === 0) {
      toast.error("No valid rows to import");
      return;
    }

    setState("importing");
    importCSV({ rows: validRows }, {
      onSuccess: (data) => {
        toast.success(`Imported ${data.importedCount} rows` +
          (data.errors.length ? ` — ${data.errors.length} errors` : ""));
        setResults([]);
        setState("idle");
      },
      onError: (error: Error) => {
        toast.error(error.message);
        setState("validated");
      }
    });
  }
}
```

**Key concept — The state machine:**

```
     ┌─────────┐
     │  idle   │◄──────────────────┐
     └────┬────┘                   │
          │ file selected          │ import success
          ▼                        │
     ┌─────────┐                   │
     │ parsing │                   │
     └────┬────┘                   │
          │ parsed + validated     │
          ▼                        │
     ┌───────────┐                 │
     │ validated │────► (clear)────┤
     └────┬──────┘                 │
          │ user clicks import     │
          ▼                        │
     ┌───────────┐                 │
     │ importing │─────────────────┘
     └───────────┘
          │ import error
          └──► back to validated
```

Four states, clear transitions. Every UI element asks "what state am I in?" rather than relying on scattered booleans (`isLoading`, `hasFile`, `hasValidated`). This scales. If you need to add a fifth state ("preview" between validated and importing), you add one enum value and one block of conditional UI.

**Key concept — `e.target.value = ""` after processing:**

HTML file inputs only fire `onChange` when the selected file changes. If the user selects `leads.csv`, sees a validation error, fixes the CSV, and tries to select the same file again — nothing fires, because "same filename" means "no change" to the browser. Resetting the input value clears the selection so the next pick is treated as new.

**Key concept — Template download via Blob + ObjectURL:**

```typescript
function handleDownloadTemplate() {
```

```typescript
  const leads = await prisma.lead.findMany({
    where,
    select: {
      phone: true,
      name: true,
      email: true,
      assignedTo: { select: { email: true } },
      stage: true,
      status: true,
      createdAt: true,
    },
    orderBy: { createdAt: "desc" },
  });

  const rows = leads.map((lead) => ({
    phone: lead.phone,
    name: lead.name,
    email: lead.email ?? "",
    assigneeEmail: lead.assignedTo?.email ?? "",
    stage: lead.stage,
    status: lead.status,
    createdAt: lead.createdAt.toISOString(),
  }));

  return buildCSVString(rows, [
    "phone", "name", "email", "assigneeEmail", "stage", "status", "createdAt",
  ]);
}
```

**Key design — role-scoped WHERE clause:**

Agents can only export their own leads. Managers and admins export everything. This is the same pattern from Session 5's activity scoping.

```typescript
export async function GET() {
```
